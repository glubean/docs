# Upload Your First Run

Use this page when local tests already work and you want a shared run history in Cloud.

## Before you start

- The CLI is installed
- `glubean run` already works locally
- You have access to a Glubean Cloud project

## 1. Create a project token

In Glubean Cloud, open your project and go to **Settings тЖТ Tokens**.

Create a token with the **`runs:write`** scope. Project tokens start with `gpt_`. Copy the value now тАФ Glubean won't show it again.

For when to use a project token vs a user API key, see [Tokens](/cloud/tokens).

## 2. Store the token

### Local development

Put the token in `.env.secrets` (must be gitignored), and the project ID in `package.json`:

```bash
# .env.secrets тАФ NOT committed
GLUBEAN_TOKEN=gpt_your_project_token
```

```json
// package.json тАФ committed; shared by every contributor
{
  "glubean": {
    "cloud": {
      "projectId": "proj_abc123"
    }
  }
}
```

Project ID isn't sensitive and belongs in the repo. Token is sensitive and goes in `.env.secrets`.

> **Working on multiple Glubean projects from one laptop?** See [Multi-workspace setup](/cloud/tokens#multi-workspace-setup).

### CI

Store the project token in your provider's secret store and inject it as the `GLUBEAN_TOKEN` env var. The CLI auto-picks it up тАФ **no `glubean login` step in CI**.

GitHub Actions example:

```yaml
- name: Upload Glubean run
  env:
    GLUBEAN_TOKEN: ${{ secrets.GLUBEAN_TOKEN }}
  run: npx glubean run --ci --upload
```

The project ID is in `package.json`, so no second secret is needed.

## 3. Upload a run

```bash
npx glubean run --ci --upload
```

You can still target a file, folder, filter, or tag:

```bash
glubean run ./tests/auth.test.ts --upload
glubean run --filter "login" --upload
glubean run --tag smoke --upload
```

## 4. Open the run

After upload, the CLI prints a Cloud URL. Open it to inspect:

- Run summary
- Test results
- Logs, traces, and metrics

## Credential lookup order

The CLI uses the first matching credential source for `GLUBEAN_TOKEN`, `GLUBEAN_PROJECT_ID`, and `GLUBEAN_API_URL`:

1. `--token` / `--project` / `--api-url` CLI flags
2. `GLUBEAN_TOKEN` / `GLUBEAN_PROJECT_ID` / `GLUBEAN_API_URL` env vars
3. `.env` and `.env.secrets` files in the project root
4. `package.json` тЖТ `glubean.cloud.token` / `.projectId` / `.apiUrl`
5. `~/.glubean/credentials.json` (written by `glubean login`)

The chain is per-field тАФ for example, the token can come from env var while the project ID comes from `package.json`. This is what makes the recommended CI pattern (1 secret) work.
