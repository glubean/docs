# Glubean

**Verification as code.** Write checks in TypeScript. Run them in editor, CI, and production. Track results over time.

[![GitHub](https://img.shields.io/badge/GitHub-glubean%2Fglubean-181717?logo=github)](https://github.com/glubean/glubean)

## What Glubean Is

Glubean is a **verification platform** — not just an API client, not just a test runner.

You write verification logic as TypeScript code: HTTP assertions, multi-step workflows, schema validation, custom metrics. That code lives in git, runs locally from your editor, executes in CI on every push, and optionally runs on a schedule against production. Results flow to a shared dashboard with analytics, alerting, and trend tracking.

```typescript
import { test } from "@glubean/sdk";

export const health = test("health-check", async (ctx) => {
  const res = await ctx.http.get(`${ctx.vars.require("BASE_URL")}/health`);
  ctx.expect(res.status).toBe(200);
  ctx.expect(res.body.version).toBeDefined();
  ctx.metric("health_latency_ms", res.timings.total, { unit: "ms" });
});
```

One file. Runs in VS Code with a click, in GitHub Actions with `glubean run --upload`, on a cron schedule via Cloud Jobs. Same code, same assertions, same traces everywhere.

## How It's Different

### vs Postman

Postman is a GUI for sending requests. Glubean is code for verifying systems.

| | Postman | Glubean |
|---|---|---|
| Authoring | Click forms, write pre/post scripts | TypeScript — full language, full IDE support |
| Version control | Export JSON, hope nobody merges wrong | `.ts` files in git, reviewed in PRs like any code |
| CI | Newman (separate tool, separate format) | Same `glubean run` command everywhere |
| Assertions | JS snippets in a text box | Fluent `ctx.expect()`, schema validation, soft/hard modes |
| Multi-step | Chained requests via collection variables | Type-safe `.setup() → .step() → .teardown()` builder |
| Data-driven | CSV runner with limited control | `test.each()`, `test.pick()`, `fromCsv()`, `fromYaml()`, directory loaders |

Postman is great for one-off exploration. Glubean is for checks you want to keep running forever.

### vs Checkly

Checkly is a monitoring SaaS — you write checks in their platform, they run them on their infrastructure, you pay per check.

Glubean inverts this:

- **Your code, your infra.** Tests run on your machine or CI runner. Cloud is the data layer — it stores results, never executes your code.
- **Local-first development.** Write in VS Code with instant feedback, traces, and debugging. Checkly's workflow starts and ends in a browser.
- **No per-check pricing.** Run as many tests as you want locally. Cloud pricing is by uploaded runs, not by check count.
- **SDK-first, not monitoring-first.** Glubean starts as a testing tool and extends to monitoring. Checkly starts as monitoring and bolts on testing.

### vs Playwright / Jest / Vitest

Those are test runners. Glubean is a verification platform that happens to include a runner.

The difference: after your tests run, Glubean gives you structured traces, custom metrics, trend analytics, flaky test detection, alerting, and a shared dashboard. A test runner gives you a green/red exit code.

Glubean also covers API + browser + MCP testing in one SDK. No glue code between Playwright for browser and Supertest for API — same `ctx`, same trace timeline.

## The Workflow

```
explore/          →  tests/           →  CI + Cloud
(scratchpad)         (committed)         (automated)
```

1. **Explore**: Write quick checks in `explore/`. Run from the editor. Iterate fast.
2. **Commit**: When a check is stable, move it to `tests/`. It's now a regression test.
3. **Automate**: CI runs `glubean run --upload` on every push. Results appear in Cloud with full traces, analytics, and alerts.
