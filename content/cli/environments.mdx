# Environments & Secrets

This is the canonical page for env files, secret files, lookup order, and `${...}` mapping.

## Design: vars and secrets are separate by architecture

Most tools mix all environment variables into one bucket. Glubean splits them into two distinct systems at the SDK level:

| | `.env` (vars) | `.env.secrets` (secrets) |
|---|---|---|
| **API** | `ctx.vars.require("KEY")` | `ctx.secrets.require("KEY")` |
| **Git** | Committed — shared baseline | Gitignored — never in repo |
| **Content** | URLs, feature flags, public IDs | API keys, tokens, passwords |
| **Cloud upload** | Visible in dashboard | Redacted by builtin rules (key-name matching) |
| **Hover preview** | Full value | Masked (`sk-l****en`) |

**Why this matters:**
- You cannot accidentally commit a secret — it's in a different file that's gitignored
- You cannot accidentally read a secret as a var — the SDK APIs are separate
- CI environments map naturally — vars from config, secrets from CI secret store
- Cloud redaction is automatic — secrets are identified by source, not by guessing

## File model

```text
.env                    # public vars — committed
.env.secrets            # credentials — gitignored
.env.staging            # staging vars — committed
.env.staging.secrets    # staging credentials — gitignored
.env.prod               # production vars — committed
.env.prod.secrets       # production credentials — gitignored
```

Add this to `.gitignore`:

```text
*.secrets
```

## What your tests read

- `ctx.vars.require("KEY")` — reads from `.env`, throws if missing
- `ctx.vars.get("KEY")` — reads from `.env`, returns `undefined` if missing
- `ctx.secrets.require("KEY")` — reads from `.env.secrets`, throws if missing

```typescript
const baseUrl = ctx.vars.require("BASE_URL");     // from .env
const token = ctx.secrets.require("API_TOKEN");    // from .env.secrets
```

**`vars` never reads `.env.secrets`. `secrets` never reads `.env`.** They are isolated by design.

### Template resolution in configure()

When you use `{{KEY}}` in `configure()` HTTP options, resolution searches **secrets first, then vars**:

```typescript
export const api = configure({
  http: {
    prefixUrl: "{{BASE_URL}}",                  // found in .env
    headers: {
      Authorization: "Bearer {{API_TOKEN}}",    // found in .env.secrets
    },
  },
});
```

> **Rule of thumb:** If leaking the value would be a security incident, put it in `.env.secrets`. Everything else goes in `.env`.

## Lookup order

Glubean resolves each key by name. File values win first, then shell or system environment is used as fallback.

- `ctx.vars.*("KEY")`: `.env` (or `--env-file`) -> system environment `KEY`
- `ctx.secrets.*("KEY")`: matching `.secrets` file -> system environment `KEY`

This means if your shell already has `BASE_URL` and `API_KEY`, you can run tests without duplicating those values into local files.

```bash
export BASE_URL=https://api.example.com
export API_KEY=sk_live_xxx
glubean run
```

If your shell variable names are different from the names used in tests, map them in `.env` / `.env.secrets` with `${...}`.

```dotenv
# .env
BASE_URL=${GLUBEAN_URL}

# .env.secrets
API_KEY=${GLUBEAN_API_KEY}
```

Then your tests can keep using:

```typescript
const baseUrl = ctx.vars.require("BASE_URL");
const apiKey = ctx.secrets.require("API_KEY");
```

## Switching environments

- Local default: `.env`
- Staging: `--env-file .env.staging`
- Production verification: `--env-file .env.prod`

```bash
glubean run ./tests --env-file .env.staging
```

The extension can switch the active env from the status bar.
