# Environments & Secrets

Glubean separates public configuration from sensitive credentials **at the SDK
level** ŌĆö not just by convention. `vars` and `secrets` are two different APIs
with two different storage files. The safe path is explicit: committed config
lives in `.env`, credentials live in gitignored `.env.secrets`.

This is different from most tools where all environment variables live in one place and security depends on discipline.

For the canonical rules (file model, resolution order), see [CLI Environments & Secrets](/cli/environments).

---

## The separation model

| | `.env` (vars) | `.env.secrets` (secrets) |
|---|---|---|
| **API** | `ctx.vars.require("KEY")` | `ctx.secrets.require("KEY")` |
| **Git** | Committed | Gitignored |
| **Content** | URLs, flags, public IDs | API keys, tokens, passwords |
| **Hover preview** | Full value shown | Masked (`sk-l****en`) |
| **Cloud upload** | Visible in dashboard | Registered for redaction by builtin rules |

## Environment files

- `.env` ŌĆö default public variables
- `.env.staging`, `.env.prod`, etc. ŌĆö named environments
- `.env.secrets` ŌĆö sensitive values for the default environment
- `.env.staging.secrets` ŌĆö sensitive values for the staging environment

`.env.secrets` files should be added to `.gitignore`.

---

## Switching environments

The active environment is shown in the status bar:

```
$(server-environment) env: default
```

Click it (or run `Glubean: Select Environment` from the Command Palette) to open the environment picker. All subsequent test runs use the selected `.env` file.

### Display names

| File | Shown as |
|---|---|
| `.env` | default |
| `.env.staging` | staging |
| `.env.prod` | prod |

### Excluded files

These files never appear in the picker:

- `.env.secrets` (and any `*.secrets` variant)
- `.env.local`
- `.env.example`

The selection persists across VS Code restarts (stored in workspace state).

## Accessing variables in tests

```typescript
import { configure, test } from "@glubean/sdk";

const { http: api } = configure({
  http: {
    prefixUrl: "{{BASE_URL}}",
    headers: { Authorization: "Bearer {{API_TOKEN}}" },
  },
});

export const getUser = test("get-user", async (ctx) => {
  const res = await api.get("users/me");
  ctx.expect(res).toHaveStatus(200);
});
```

For full API details (`get` vs `require`, validation), see [SDK Environment & Secrets](/sdk/test/env-and-secrets).

## Template resolution in configure()

When you use `{{KEY}}` templates in `configure()` HTTP options (headers, prefixUrl), the resolution searches **secrets first, then vars**:

```typescript
export const { vars, secrets, http: api } = configure({
  vars: { baseUrl: "{{BASE_URL}}" },
  secrets: { token: "{{API_TOKEN}}" },
  http: {
    prefixUrl: "{{BASE_URL}}",                    // found in .env (vars)
    headers: {
      Authorization: "Bearer {{API_TOKEN}}",      // found in .env.secrets (secrets)
    },
  },
});
```

This means you _can_ put a secret in `.env` and `{{KEY}}` will still find it ŌĆö
**but you shouldn't.** Keep secrets in `.env.secrets` so they stay gitignored,
masked in hover preview, and registered for redaction on Cloud upload.

Redaction is a safety layer, not a reason to print credentials. Do not log
tokens, cookies, signed URLs, or derived credential strings from test code.

> **Where to put what:** If the value would be dangerous to leak in a git commit or a dashboard screenshot, it belongs in `.env.secrets`. Everything else goes in `.env`.

## Hover preview

Hover over `vars.require("KEY")` or `secrets.require("KEY")` in your test file to see the resolved value from the currently selected environment.

- **vars** ŌĆö shows the full value and its source file:
  ```
  BASE_URL = "https://api.staging.example.com"  (.env.staging)
  ```
- **secrets** ŌĆö value is masked. Only the first 4 and last 2 characters are shown, the rest is replaced with `****`:
  ```
  API_TOKEN = "sk-l****en"  (.env.staging.secrets)
  ```
- **Not found** ŌĆö shows a warning:
  ```
  WARNING  BASE_URL ŌĆö not found in .env.staging
  ```
