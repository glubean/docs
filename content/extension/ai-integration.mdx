# AI Context Guide

Glubean achieves reliable AI test generation through three context layers: **OpenAPI specs** (what the API looks like), **Skills** (how to write tests), and **MCP tools** (run and validate). Each layer is useful alone. Together, they produce tests that run on first try.

---

## OpenAPI Specs

The fastest way to get accurate tests is to give AI your API specification. Instead of guessing endpoints and response shapes, the AI reads them directly from your spec.

### Providing your spec

Drop your OpenAPI spec into the `context/` folder:

```
my-project/
├── context/
│   ├── openapi.json          # or openapi.yaml
│   └── openapi.patch.yaml    # optional: supplement missing schemas
├── tests/
├── explore/
└── deno.json
```

The AI reads `context/` before generating any test. Supported formats: OpenAPI 3.x in JSON or YAML.

### AI-ready annotations

Not all OpenAPI specs are equal. A spec with detailed annotations — response shapes, error codes, parameter constraints, and `operationId` on every endpoint — gives dramatically better results than a minimal spec with just paths and methods.

When building your own API with Glubean Cloud, annotations are added automatically. For external APIs, you can supplement incomplete specs with a **patch file**:

```yaml
# context/openapi.patch.yaml
endpoints:
  /users/{id}:
    get:
      200:
        id: string
        name: string
        email: string
      404:
        error: '"not_found"'
```

Apply with:

```bash
glubean patch context/openapi.json --patch context/openapi.patch.yaml
```

### Splitting large specs

If your spec has more than ~50 endpoints, split it into per-endpoint files so the AI only reads what it needs:

```bash
glubean spec split context/openapi.json
```

This creates `context/openapi-endpoints/_index.md` (a summary the AI reads first) plus individual endpoint files. The AI opens only the specific endpoint file it needs, keeping context small and focused.

### Dual specs

If your API has both public and internal endpoints, generate separate specs. Glubean Cloud does this automatically:

- `openapi-open.json` — public endpoints only (what SDK users see)
- `openapi-internal.json` — all endpoints (for internal tooling)

This prevents AI from generating tests against internal-only endpoints when you only want public API coverage.

## Skills

A **skill file** teaches your AI assistant the exact Glubean SDK API, your project conventions, and the patterns your team follows. It is the single biggest factor in generation quality.

### What a skill is

A skill is a markdown file with YAML frontmatter. It contains:

- The complete SDK API reference (`test()`, `ctx.http`, `ctx.expect`, `configure()`, etc.)
- Code patterns for common scenarios (CRUD, auth flows, error tests)
- Project rules (naming conventions, tag policy, cleanup requirements)
- Anti-patterns to avoid (no raw `fetch`, no hardcoded secrets, no guessing endpoints)
- Allowed MCP tools the AI can use

### Where skills live

| AI Assistant | Location |
|---|---|
| **Claude Code** | `.claude/rules/gb.md` |
| **Cursor** | `.cursor/rules/glubean.mdc` |
| **Windsurf** | `.windsurfrules` |

`glubean init` generates the skill file automatically for your editor. You can customize it to add team-specific patterns.

### What the skill locks

Without a skill file, AI assistants generate _roughly_ correct code — right idea, wrong imports, inconsistent style, missing cleanup. With a skill file:

| Aspect | Without skill | With skill |
|---|---|---|
| Imports | `import { test } from "jsr:@glubean/sdk"` | `import { test } from "@glubean/sdk"` (alias) |
| Secrets | `const token = process.env.TOKEN` | `ctx.secrets.require("TOKEN")` |
| Auth | Inline header construction | `configure()` with `{{TOKEN}}` interpolation |
| Cleanup | None | `.teardown()` on every data-creating test |
| Assertions | `if (res.status !== 200) throw` | `ctx.expect(res).toHaveStatus(200)` |
| Coverage | Happy path only | Success + 401 + 400 + 404 boundaries |

The skill is the difference between "looks right" and "runs right."

<Callout type="info">
**You don't need to write skills from scratch.** `glubean init` generates a comprehensive skill file based on the current SDK version. Customize it by adding your team's auth patterns, naming conventions, or assertion policies.
</Callout>

## MCP Tools

**MCP (Model Context Protocol)** closes the generation loop. Instead of the AI writing code and hoping it works, MCP tools let the AI _run_ the test, _read_ structured failures, _fix_ the code, and _rerun_ — all within one chat conversation.

### The closed loop

<Steps>

### AI writes the test
Using your OpenAPI spec and skill file, the AI generates a test file.

### AI runs via MCP
The AI calls `glubean_run_local_file` to execute the test locally. No manual copy-paste or terminal switching needed.

### AI reads structured failures
MCP returns structured output — not raw console logs, but typed assertion results, HTTP traces, and error messages the AI can parse reliably.

### AI fixes and reruns
The AI reads the failure, edits the code, and calls `glubean_run_local_file` again. This loop repeats until all assertions pass.

</Steps>

This is fundamentally different from "generate and pray." The AI validates its own output.

### Configuration

`@glubean/mcp` is a stdio-based MCP server. Configure it for your editor:

<Tabs items={['Cursor', 'VS Code (Copilot)', 'Claude Code']}>

<Tabs.Tab>

Create `.cursor/mcp.json` in your project root:

```json
{
  "mcpServers": {
    "glubean": {
      "command": "deno",
      "args": ["run", "-A", "jsr:@glubean/mcp"]
    }
  }
}
```

Also add a rule file at `.cursor/rules/glubean.mdc` to improve tool selection:

```markdown
---
description: "Glubean API testing — run, debug, create, fix tests"
alwaysApply: true
---

When the user asks to run, debug, create, or fix API tests, use the Glubean MCP tools:
- Run a test file → glubean_run_local_file
- List test files → glubean_list_test_files
- Inspect exports → glubean_discover_tests
- Check config → glubean_diagnose_config
- Review last run → glubean_get_last_run_summary
```

</Tabs.Tab>

<Tabs.Tab>

Create `.vscode/mcp.json` in your project root:

```json
{
  "servers": {
    "glubean": {
      "type": "stdio",
      "command": "deno",
      "args": ["run", "-A", "jsr:@glubean/mcp"]
    }
  }
}
```

</Tabs.Tab>

<Tabs.Tab>

If `@glubean/mcp` is installed, Claude Code discovers it automatically. No manual configuration needed.

The skill file (`.claude/rules/gb.md`) already declares allowed MCP tools in its frontmatter.

</Tabs.Tab>

</Tabs>

### Key tools

| Tool | What it does |
|---|---|
| `glubean_run_local_file` | Execute a test file and return structured results (assertions, traces, logs) |
| `glubean_discover_tests` | List all `test` exports in a file with their names and tags |
| `glubean_list_test_files` | Find all test files in the project |
| `glubean_diagnose_config` | Check project health (deno.json, .env, directories) and suggest fixes |
| `glubean_get_last_run_summary` | Summary of the most recent local run |
| `glubean_get_local_events` | Detailed events from last run — filter by type (assertion, trace, log) |

<Callout type="warning">
**If the AI ignores MCP tools:** mention "glubean" explicitly in your prompt — e.g., "use glubean to run the test" instead of just "run the test." AI assistants match tools by keyword, and a specific mention dramatically improves hit rate.
</Callout>

## The Combination

Each layer solves a different failure mode:

| Layer | What it prevents |
|---|---|
| **OpenAPI** | Guessing endpoints, wrong request shapes, missing error codes |
| **Skills** | Wrong SDK API, inconsistent style, missing cleanup, shallow coverage |
| **MCP** | "Looks correct but actually fails" — validates before you review |

Without any context, AI generates plausible-looking code that usually needs 2-3 rounds of manual fixes. With all three layers, generated tests follow your patterns, target your real API, and are verified before you see the diff.

## Quick Setup

<Steps>

### Initialize project

```bash
glubean init
```

This creates the project structure, skill file, and MCP config for your editor.

### Add your API spec

Drop your OpenAPI spec into `context/`. For large specs:

```bash
glubean spec split context/openapi.json
```

### Configure MCP

Follow the [editor-specific instructions above](#configuration).

### Generate your first test

Ask your AI assistant:

> "Read the OpenAPI spec in context/ and generate a test for the checkout endpoint. Run it and fix any failures."

The AI reads the spec, follows the skill conventions, generates the test, runs it via MCP, and iterates until it passes.

</Steps>
