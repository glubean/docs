# Running Tests

This page covers the editor-side ways to run Glubean checks.

## Run from the gutter

Every `test()` call gets a **▶** button in the gutter. Use it to run one test immediately.

When the run finishes, Glubean updates the gutter state and opens the trace viewer beside your code.

## Run the whole file

When a `*.test.ts` file is open, the editor title bar also exposes a run button for the entire file.

## Test Explorer

Glubean integrates with VS Code's native Test Explorer.

You usually see two groups:

- **Explore** for scratchpad checks in `explore/`
- **Tests** for committed checks in `tests/`

From Test Explorer you can run a folder, file, or single test and inspect the latest pass or fail state.

### Multi-step tests

Builder-style tests appear as trees so you can see which step failed:

```text
▼ auth-flow
  ✓ login          (142ms)
  ✓ get profile    (89ms)
  ✗ update name    AssertionError: expected 200, got 422
```

## Re-run the previous target

Use the rerun shortcut for the most recent target:
- **Mac:** `Cmd + Shift + R`
- **Windows/Linux:** `Ctrl + Shift + R`

## Data-driven tests

`test.each()` rows appear as child nodes:

```typescript
export const statusCodes = test.each([
  { id: 1,   expected: 200 },
  { id: 999, expected: 404 },
])("get-item-$id", async (ctx, { id, expected }) => {
  const res = await ctx.http.get(`/items/${id}`);
  ctx.expect(res.status).toBe(expected);
});
```

You can run a single row from the tree.

`test.pick()` examples show up as CodeLens actions above the test so you can run `normal`, `edge`, or other named examples directly.

## Tasks Panel

If your team defines `glubean run` commands in `package.json` scripts, the Tasks Panel turns them into clickable presets such as `test:staging` or `test:smoke`.

This is useful when QA or non-authors need stable run buttons without learning CLI flags.
